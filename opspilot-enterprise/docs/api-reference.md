## Authentication

### POST /api/v1/auth/login

Login with username/password. Returns JWT in HttpOnly cookie `opspilot_token`.

**Body:** `{ "username": "admin", "password": "admin123" }`

**Response data:** `{ "username", "display_name", "role", "avatar", "token" }`

### GET /api/v1/auth/me

Get current authenticated user. Returns 401 if not authenticated.

### POST /api/v1/auth/logout

Clears the authentication cookie.

## Chat & Diagnosis (Enhanced)

### GET /api/v1/chat/sessions

List all chat sessions, sorted by `updated_at` descending.

### GET /api/v1/chat/sessions/{id}/messages

List all messages in a session, including user and assistant messages with tool traces and evidence refs.

### POST /api/v1/chat/sessions/{id}/messages

Send a message. If diagnosis-intent keywords are detected (e.g. "analyze", "diagnose", "root cause"), the orchestrator returns structured diagnosis results including:
- `diagnosis_id`, `root_cause_candidates`, `evidence_refs`, `tool_traces`, `recommended_actions`

### GET /api/v1/chat/sessions/{id}/evidence

Session-level evidence list, populated after diagnosis-triggering messages.

### GET /api/v1/chat/sessions/{id}/tool-traces

Session-level tool call trace list.

### GET /api/v1/chat/diagnoses/{diagnosis_id}

Retrieve a specific diagnosis result by ID, including all evidence, tool traces, and root cause candidates.

## LLM Integration

The Orchestrator integrates with Zhipu AI (GLM-5-turbo) via OpenAI-compatible API.

- **General chat**: Messages are sent to the LLM with an ops-expert system prompt and conversation history
- **Diagnosis mode**: When diagnosis keywords are detected, the LLM generates analysis based on collected evidence using a specialized RCA system prompt
- **Fallback**: If LLM is unavailable (`LLM_ENABLED=false` or API error), responses degrade gracefully to mock/template replies
- **Configuration**: Set via environment variables `LLM_API_BASE`, `LLM_API_KEY`, `LLM_MODEL` (see `.env.example`)
